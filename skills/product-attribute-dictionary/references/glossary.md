# Glossary: Product Attribute Dictionary

## Overview

One of the inputs for this skill is a Product Attribute Dictionary, produced by the Map Your Product Attribute Dictionary skill. This glossary explains how that document is structured: what each section contains, what values to expect, and how to interpret them.

**Producing skill:** Map Your Product Attribute Dictionary
**Current version:** v0.1
**Document type:** Product Attribute Dictionary
**Purpose:** Defines the structure of a brand's product catalog so downstream skills know what fields exist, what values they accept, and how variants work for each product type.
**Consumers:** Skills that write product content, audit catalog data, optimize product feeds, generate bulk CSV operations, or map products to external taxonomies.

## Document Structure

| Section | Description |
|---|---|
| Overview | Platform, category count, product type count, and whether metafield data was included. |
| Standard Fields | Every column in the standard product export, with format and description. |
| Metafields | Every custom field with namespace, key, format, and description. May be a placeholder if metafield data was not provided. |
| Product Types | Table of every product type, its category, and which variant dimensions apply. |
| Variant Attributes | How each variant dimension works: format, scale, range, product-type-specific behavior. |
| Product Type Profiles | Per-type or per-group view: which metafields apply, how the variant matrix works. |
| Conventions | Catalog-wide structural patterns: SKU encoding, tag taxonomy, image positions, naming conventions. |

## Section Hierarchy

When multiple sections define the same field or provide overlapping guidance:

1. Product Type Profiles (most specific). If a type profile specifies something about a field that differs from the catalog-wide definition, follow the type profile.
2. Variant Attributes (dimension-specific). Defines how each dimension works per product type.
3. Metafields and Standard Fields (catalog-wide definitions). Apply when no type-specific definition exists.
4. Conventions (structural patterns). Apply across all fields and types. Do not override specific field definitions.
5. Overview (contextual). General catalog context, not a source of field-level definitions.

## Handling Missing or Unexpected Input

**The Metafields section contains placeholder text:**
Metafield data was not provided when the dictionary was produced. Do not invent metafield definitions. Work only with Standard Fields and Variant Attributes. The placeholder may list probable metafields; these are suggestions, not confirmed definitions.

**An expected section is missing entirely:**
Treat the same as placeholder text. Produce output using the fields that are documented.

**The version marker is older than the current glossary version:**
Check the changelog. Apply current definitions, but account for structural differences noted for that version.

**The document was not generated by this skill:**
The user may have written their own attribute dictionary. Do not reject it. Extract whatever field definitions are present, map them to the fields you understand, and produce output. Fall back to platform defaults where coverage is missing.

## Field Definitions

### Overview

**What it represents:** Context for the catalog's structure and scope.
**What to expect:** A paragraph identifying the platform, categories, product type count, and metafield inclusion status.
**How to apply:** Use to understand which platform conventions apply. Not a source of field-level definitions.

### Standard Fields

**What it represents:** Catalog-wide column definitions for the standard product export.
**What to expect:** Table with one row per column. Each row includes format (data type and structure) and a description of purpose and conventions.
**How to apply:** When you need to read, write, or validate a column from a product export, look it up here. The format tells you the data type. The description tells you what values to expect and any conventions.

#### Format Vocabulary

| Value | Definition |
|---|---|
| Free text | No structural constraints. Unique or near-unique per product. |
| Structured string | Follows a defined pattern (described in the field's description or in Conventions). |
| Controlled list | Values come from a defined set (described in the field's description). |
| Comma-separated list | Multiple values in a single field, separated by commas. |
| HTML | Content with HTML markup. |
| URL | Full web address. |
| Decimal | Numeric with decimal places. Precision noted in description (e.g., "two places"). |
| Integer | Whole number. |
| Boolean | true/false. |
| String | Short text, often from a small set of values noted in the description. |
| Lowercase hyphenated slug | URL-safe identifier (e.g., "cascade-rain-shell"). |
| Two-letter code | Standardized code (e.g., ISO 3166-1 alpha-2 country codes). |
| Varies | Format depends on context (e.g., variant option values that differ by product type). See the referenced section for details. |

### Metafields

**What it represents:** Custom field definitions not included in the standard export.
**What to expect:** Table with namespace, key, format, and description per field. Some fields note that they only apply to certain product types.
**How to apply:** Same as Standard Fields. Look up a metafield by key. If a metafield's description says it only applies to certain types, do not expect it on other types. If the section is a placeholder, do not assume metafield definitions.

### Product Types

**What it represents:** Complete list of product types, their categories, and variant dimensions.
**What to expect:** Table with category, product type name, and applicable variant dimensions.
**How to apply:** Use to determine which product types exist and which variant dimensions they use. When your task varies by product type, start here to understand the scope.

### Variant Attributes

**What it represents:** How each variant dimension works across the catalog.
**What to expect:** One entry per dimension as prose. Describes the format, scale, range, and any product-type-specific behavior. Current values may appear as examples but should not be treated as the complete valid set.
**How to apply:** When generating or validating variant data, match the documented conventions exactly. If the dictionary says apparel sizes use abbreviated letter format (S, M, L, XL), generate those, not "Small, Medium, Large, Extra Large."

### Product Type Profiles

**What it represents:** Type-specific or group-specific view of which fields apply and how variants work.
**What to expect:** Each profile lists which metafields apply to that type (all vs. some), and describes the variant matrix structure (which dimensions combine, typical matrix size, any incomplete matrix behavior).
**How to apply:** When working with a specific product type, read its profile first. It tells you which metafields to expect and how the variant structure works. Then refer to the Metafields and Variant Attributes sections for the full definitions of those fields.

### Conventions

**What it represents:** Catalog-wide structural patterns that apply across multiple fields and types.
**What to expect:** Named entries for SKU encoding, tag taxonomy, image positions, naming conventions, gender handling, or other structural decisions.
**How to apply:** Follow when generating new data or validating existing data. Conventions describe how the catalog is structured. Match the documented patterns.
